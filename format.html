Output Format

Tensor bundles with canonical JSON metadata.

TensorSuite stores each tensor as a folder. The *_metadata.json file is the canonical schema for tensor attributes and metadata; .tns and .tnsb are data representations inside the bundle.

Element-wise

Element-wise tensor format.

Element-wise tensors store each non-zero entry as explicit coordinates followed by one value.

Bundle Layout

One folder per tensor record.

A bundle includes a tensor-name metadata file, data files, a README, and potentially additional files the dataset needs.

Named Files

A tensor-name folder uses matching file prefixes, such as example-tensor_metadata.json, example-tensor.tns, and example-tensor.tnsb.

Metadata

*_metadata.json records identity, attributes, tensor size information, and file mappings for the tensor folder.

Text Tensor

.tns stores coordinate-based tensor entries in a readable text format for inspection.

Binary Tensor

.tnsb stores coordinate-based tensor entries in binary form for faster loading. Large bundles may provide only .tnsb as the tensor data file, with no .tns file. 

example-tensor/
  example-tensor_metadata.json
  example-tensor.tns
  example-tensor.tnsb
  README.md
large-tensor/
  large-tensor_metadata.json
  large-tensor.tnsb
  README.md

Tensor Data Files

Coordinate data in text and binary files.

The text file keeps a minimal cross-check header, one size line, and coordinate entries. Full attributes and metadata stay in the bundle metadata JSON.

Minimal Header

The first line is %%TensorSuite-TNS. Then % version and % name let readers cross-check the file against the metadata JSON.

Size Line

One line gives the tensor order, all mode dimensions, and NNZ so readers can preallocate memory and validate the entry count.

Entries

Each data row stores one-based or zero-based coordinates followed by a non-zero value. 

%%TensorSuite-TNS
% version: 0.1
% name: example-tensor
3 4 3 2 5
1 1 1 1.0000000000000000
1 2 1 2.5000000000000000
2 2 1 -1.2500000000000000
3 1 2 4.0000000000000000
4 3 2 0.5000000000000000
%%TensorSuite-TNSB
% version: 0.1
% name: example-tensor
3 4 3 2 5
[binary coordinate/value records begin here]

Metadata Information

Metadata .json file description.

The JSON file stays comment-free; attribute meanings and allowed values are documented below. Auxiliary data can be documented in README.md. 

{
  "version": "0.1",
  "name": "example-tensor",
  "group": "TensorSuite-example",
  "id": "0001",
  "time": "2026-05-06",
  "source": "generated",
  "source_url": "",
  "value_type": "float64",
  "value_domain": "general",
  "index_type": "uint64",
  "index_base": 1,
  "sorted": "lexicographic",
  "sort_order": [1, 2, 3],
  "duplicates": "disallowed",
  "explicit_zeros": "allowed",
  "pattern_symmetry": "no",
  "numerical_symmetry": "no",
  "sparsity_type": "element",
  "dense_modes": [],
  "endianness": "little",
  "order": 3,
  "dimensions": [4, 3, 2],
  "nnz": 5,
  "files": {
    "text": "example-tensor.tns",
    "binary": "example-tensor.tnsb",
    "readme": "README.md"
  }
}
Attribute Allowed Values Meaning
value_type float32, float64, int32, int64 Numeric type used to store tensor values.
value_domain general, nonnegative, binary Numerical value constraint. nonnegative means values are greater than or equal to zero; binary means values are restricted to 0 and 1.
index_type uint32, uint64 Integer type used by binary coordinate indices.
endianness none, little, big Byte order used by binary numeric records. Use none when the bundle has no binary tensor data.
index_base 0, 1 Whether coordinates are zero-based or one-based.
sorted none, lexicographic Whether entries are sorted.
sort_order Permutation of mode ids, such as [1,2,3] Mode priority used for lexicographic sorting.
duplicates allowed, disallowed, summed Policy for repeated coordinates.
explicit_zeros allowed, disallowed Whether stored entries may have numerical value zero.
pattern_symmetry full, partial, no Symmetry of the nonzero coordinate pattern.
numerical_symmetry full, partial, no Symmetry of both coordinate pattern and mirrored values.
sparsity_type no, semi, element, block Dense, semi-sparse, element-sparse, or block-sparse structure.
dense_modes Mode-id list, such as [] or [1,2] Modes that are dense or nearly dense.
nnz_block Nonnegative integer, or null Number of nonzero blocks. Use null for tensors without block-sparse structure.

Block-wise

Block-wise tensor format.

Block-wise tensors use the same TensorSuite folder structure, metadata JSON, and coordinate-oriented data model. The recommended text suffix is .btns, and the recommended binary suffix is .btnsb, so block-wise files are distinct from element-wise .tns and .tnsb files. Use .btnsb directly for binary block-wise files instead of a double suffix such as .btns.bin.

Block Header

Keep one line for the full tensor: 3 2000 2000 20 nnz, followed by one line for the block grid: 3 110 110 19 nnz_block.

Block Sizes

After the two size lines, write one block-size array per mode. For a 110 x 110 x 19 block grid, the three arrays have lengths 110, 110, and 19.

Implicit Values

If a tensor has dense modes or block-sparse structure, the data file may omit values. Users can fill values according to metadata constraints and any README notes. 

{
  "version": "0.1",
  "name": "block-example",
  "group": "TensorSuite-example",
  "id": "0002",
  "time": "2026-05-06",
  "source": "generated",
  "source_url": "",
  "value_type": "float64",
  "value_domain": "nonnegative",
  "index_type": "uint64",
  "index_base": 1,
  "sorted": "lexicographic",
  "sort_order": [1, 2, 3],
  "duplicates": "disallowed",
  "explicit_zeros": "disallowed",
  "pattern_symmetry": "no",
  "numerical_symmetry": "no",
  "sparsity_type": "block",
  "dense_modes": [],
  "endianness": "little",
  "order": 3,
  "dimensions": [2000, 2000, 20],
  "nnz": 768000,
  "block_dimensions": [110, 110, 19],

    "text": "block-example.btns",
